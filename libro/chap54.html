Taming the State

Los capítulos anteriores han demostrado que la administración de estado podes ser un tema crucial en tu aplicación. En general, no sólo React, pero una gran cantidad de frameworks de SPA luchar con ella. Las aplicaciones se hicieron más compleja en los últimos años. Un gran reto en las aplicaciones web hoy en día es de dominar y controlar el estado.

The previous chapters have shown you that state management can be a crucial topic in your application. In general, not only React but a lot of SPA frameworks struggle with it. Applications got more complex in the recent years. One big challenge in web applications nowadays is to tame and control the state.

En comparación con otras soluciones yReact dado un gran paso hacia adelante. El flujo de datos unidireccional y una API simple para administrar el estado en un componente son indispensables. Estos conceptos hacen que sea más fácil de razonar acerca de su estado y tus cambios de estado. Esto hace que sea más fácil razonar sobre ello en un nivel de componentes y hasta cierto punto en un nivel de aplicación.

Compared to other solutions React already made a big step forward. The unidirectional data flow and a simple API to manage state in a component are indispensable. These concepts make it easier to reason about your state and your state changes. It makes it easier to reason about it on a component level and to a certain degree on a application level.

En una aplicación cada vez mayor se vuelve más difícil de razonar acerca de los cambios de estado. podes introducir errores mediante una operación en estado rancio cuando se utiliza un objeto más de una función en setState(). Vos tiene que levantar el estado en torno a compartir necesario o innecesario ocultar el estado a través de los componentes. podes suceder que un componente necesario para levantar el estado, debido a su componente de hermanos depende de ello. Quizás el componente está muy lejos del árbol de componentes y tiene que compartir el estado a través de todo el árbol de componentes. Sucede que los componentes se involucren en mayor medida en la gestión de estado. Pero después de todo, la responsabilidad principal de los componentes debes ser que representa la interfaz de usuario, no?

In a growing application it gets harder to reason about state changes. You can introduce bugs by operating on stale state when using an object over a function in setState(). You have to lift state around to share necessary or hide unnecessary state across components. It can happen that a component needs to lift up state, because its sibling component depends on it. Perhaps the component is far away in the component tree and you have to share the state across the whole component tree. It happens that components get involved to a greater extent in state management. But after all the main responsibility of components should be representing the UI, shouldn’t it?

Debido a todas estas razones, existen soluciones independientes para hacerse cargo de la state management. Estas soluciones no sólo se utilizan en React. Sin embargo, eso es lo que hace que el ecosistemReact un lugar tan poderosa. Se pueden utilizar diferentes soluciones para resolver tus problemas. Para abordar el problema de escalar la state management, es posible que haya oído hablar de las bibliotecas Redux o MOBX. podes utilizar cualquiera de estas soluciones en una aplicación React. Vienen con extensiones, React-redux y MOBX-React , para integrarse con React.

Because of all these reasons, there exist standalone solutions to take care of the state management. These solutions are not only used in React. However, that’s what makes the React ecosystem such a powerful place. You can use different solutions to solve your problems. To address the problem of scaling state management, you might have heard of the libraries Redux or MobX. You can use either of these solutions in React application. They come with extensions, react-redux and mobx-react, to integrate with React.
